## Why

RedShell currently quits the entire process when the user clicks the Windows close button (top-right `X`), forcing a full re-launch every time the user wants to come back. As an agent-management dashboard the user typically keeps open in the background, the app should be able to stay resident in the system tray and be reopened with a single click. The user also wants control over which behavior the close button performs (real exit vs. minimize-to-tray) without having to dig into a settings page, so the choice surfaces both as a one-time prompt on first close and as a persistent toggle on the tray icon's context menu.

## What Changes

- Add a system tray icon (Windows-first) that shows the RedShell logo, supports left-click to show/hide the main window, and exposes a right-click context menu.
- Intercept the main window's close event (`OnBeforeClose`). When the user-preferred close behavior is `minimize-to-tray`, hide the window instead of quitting; when it is `exit`, let the app close normally.
- On the first close-button click after install, show a modal asking the user to choose between "Exit RedShell" and "Minimize to tray". The chosen behavior is persisted and the modal does not appear again.
- Tray icon context menu contains:
  - **Show RedShell** — restores and focuses the main window.
  - **Close button minimizes to tray** — a checkable item that toggles the persisted close-button behavior between `minimize-to-tray` and `exit`.
  - **Quit RedShell** — bypass the close-behavior preference and exit the process.
- Persist the user's preference in `~/.redshell/preferences.json` under a new `closeBehavior` field with values `"exit"`, `"minimize-to-tray"`, or `"unset"` (default before first prompt).
- Expose the tray lifecycle and preference read/write through a new `app/AppPreferencesApp` Wails binding so the frontend modal can save the user's choice.
- Tray + close-intercept implementation lives behind `_windows` build tags; non-Windows builds compile a no-op stub so `wails build` still succeeds on macOS / Linux while the feature is Windows-only for now.

## Capabilities

### New Capabilities

- `system-tray`: Windows system tray icon, context menu, show/hide window behavior, and the close-button intercept that consults the persisted `closeBehavior` preference and the first-run prompt.
- `app-preferences`: Application-level user preferences stored in `~/.redshell/preferences.json` (currently just `closeBehavior`), with a Go service and a Wails-bound app wrapper so the frontend can read and update them.

### Modified Capabilities

- `wails-app-shell`: The shell now registers an `OnBeforeClose` hook and integrates a tray-icon lifecycle (start on app launch, stop on real exit). Closing the main window may hide it instead of quitting, depending on the persisted preference.

## Impact

- **Go backend**:
  - New `internal/preferences/service.go` (+ `service_test.go`) reading/writing `~/.redshell/preferences.json`.
  - New `internal/tray/` package: `tray_windows.go` (real implementation using `github.com/getlantern/systray`), `tray_other.go` (no-op stub) wired by `// +build` tags. Owns the icon lifecycle and the menu-item callbacks.
  - New `app/preferences.go` exposing `GetCloseBehavior`, `SetCloseBehavior`, and emitting a `tray:preference-changed` runtime event so the tray menu's checkbox state stays in sync if the frontend changes the value.
  - `main.go`: wire the new service + app wrapper, register the tray lifecycle in `OnStartup` / `OnShutdown`, and add `OnBeforeClose` that consults the preference.
- **Frontend**:
  - New `frontend/src/components/system/CloseBehaviorPromptModal.vue` rendered at the layout level. Listens for a `tray:close-behavior-prompt` Wails event and opens an `AppModal` with two buttons.
  - New `frontend/src/stores/preferences.ts` proxying the new bindings.
  - `frontend/src/App.vue` (or `DefaultLayout.vue`) mounts the modal once at app root.
  - `frontend/wailsjs/go/app/AppPreferencesApp.{ts,js}` is regenerated by Wails — do not hand-edit.
- **Dependencies**: add `github.com/getlantern/systray` to `go.mod`. The package pulls in a small CGO-free Windows shim and a CGO-required GTK shim on Linux; we guard the import with build tags so non-Windows builds stay CGO-free.
- **Assets**: ship a `.ico` for the tray (256x256 multi-resolution) at `build/windows/icon.ico` (already present for the executable) and a smaller 16x16/32x32 variant for the tray menu.
- **Settings storage**: introduces `~/.redshell/preferences.json`. Existing `~/.redshell/settings.json` (managed by `agent.SettingsService`) is untouched to keep agent config and shell preferences separable.
- **Tests**: Go unit tests for `internal/preferences` (read missing file, round-trip, malformed file). Tray code is excluded from unit tests because driving a real tray requires a desktop session; a thin interface lets the close-intercept logic be tested with a fake.
- **Docs**: `CLAUDE.md` Project Overview gains a one-liner mentioning the tray, and a section under "Agent-specific paths" gains a new row covering `preferences.json`.
